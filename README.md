## 🗂️ Структура одного запису

Кожен файл має однакову структуру розділів:

```
# Категорія N, тест ...: <Сервіс> <Модель> (запуск K)

## Метадані експерименту      → таблиця: сервіс, модель, дата, інтерфейс, запуск
## Промпт                     → дослівний текст запиту
## Вхідне зображення          → опис вхідних даних (лише для категорії 1)
## Відповідь моделі           → повна дослівна відповідь сервісу
## Оцінка експерта за рубрикою → таблиця К1–К5 та підсумкова оцінка Q
## Коментар експерта          → пояснення виставлених балів
```

Підсумкова оцінка обчислюється за формулою:

```
Q = 0,30·К1 + 0,25·К2 + 0,15·К3 + 0,20·К4 + 0,10·К5
```

де К1 — точність ідентифікації, К2 — фактична коректність, К3 — повнота, К4 — відтворюваність (на серію з 3 запусків), К5 — педагогічна цінність.

## 🖱️ Інструкція для користувача

1. **Оберіть категорію задачі**:
   - `transistor-test-*` — розпізнавання технічної схеми
   - `test-*` — генерація програмного коду

2. **Оберіть сервіс і номер запуску** у назві файлу (наприклад, `test-2-gemini.md` — Gemini, запуск 2).

3. **Прочитайте запис зверху вниз**: метадані → промпт → відповідь → оцінка → коментар.

4. **Для порівняння сервісів** відкрийте три файли одного тесту (запуски 1–3) поруч.

## 🧪 Нотатки щодо відтворюваності

| Особливість | Пояснення |
|-------------|-----------|
| Розбіжності між запусками | Генерування стохастичне — той самий промпт може дати дещо відмінні відповіді; саме тому виконано по 3 повторення |
| Латинські позначення на схемі | Використано літери B, C, E, щоб уникнути кириличної плутанини й не спотворити оцінку розпізнавання |
| Чистий діалог на кожен запуск | Кожен тест виконувався в окремому діалозі для незалежності результатів |
| Фіксація версій моделей | Версію сервісу зафіксовано в метаданих кожного запису |
